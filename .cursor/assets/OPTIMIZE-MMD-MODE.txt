## Mission
Take the existing markdown or Mermaid and improve it into a stronger architecture representation without breaking the original design intent.

This is not a rewrite-for-style task.
This is a structure-preserving optimization task.

## Required operating behavior
1. First understand what the current source is trying to represent.
2. Determine whether it is:
   - mostly correct but flat
   - mostly correct but cluttered
   - structurally weak
   - invalid Mermaid
   - under-labeled
   - overloaded and needing multiple diagrams
3. Preserve:
   - entities
   - directionality
   - architecture intent
   - key failure paths
   - important labels and semantics
4. Improve:
   - decomposition
   - readability
   - boundary clarity
   - Mermaid correctness
   - node/edge discipline
   - operational realism
5. If the current diagram should be split into multiple views, do so deliberately rather than forcing everything into one overloaded graph.
6. Render repeatedly in cheap/default mode to compare versions.
7. Assess returned diagrams critically.
8. Only use Max mode for the final optimized version once the architecture is already strong.

## Optimization rules
- Do not introduce breaking conceptual changes.
- Do not change the system into a different architecture.
- Do not add entities unless strongly implied or necessary for correctness.
- Do not flatten layered systems.
- Do not turn a valid focused diagram into a noisy “bigger” one.
- Prefer structural truth over decorative complexity.

## System capabilities (active in pipeline)
The system has the following capabilities that operate automatically during inference:
- **Multi-role planning:** Up to 3 domain-expert agents reason concurrently on optimization strategy. The best plan wins by combined quality + completeness score.
- **Graduated refinement:** Refinement pressure is gated by quantitative scoring — heavy rewrite below 0.5, targeted gap-fill between 0.5-0.7, skip above 0.7.
- **2-branch composition:** The render pipeline generates two structurally competing Mermaid compositions in parallel (TB/boundary vs LR/data-flow), scores both, and keeps the stronger candidate. For optimization, this means two alternative improvement paths compete.
- **GoT-style merge:** If both composition branches survive pruning, compatible entities from the runner-up are merged into the winner when doing so does not decrease the score.
- **Per-stage model routing:** Structured extraction uses fast models, creative composition uses worker models, and orchestration uses the orchestrator tier.
- **Render audit events:** The full render pipeline emits granular audit events for debugging and narration.
- **State budget enforcement:** The GoT controller tracks state count against a budget (default 40) and halts if exceeded.

Leverage these capabilities: the 2-branch composition is especially valuable for optimization — one branch can preserve the original layout direction while the other explores an alternative. The scorer determines which produces a better result.

## Judgment standard
A successful optimization:
- preserves the original design intent
- improves readability and structure
- improves correctness or Mermaid discipline
- improves engineering usefulness
- does not introduce unnecessary changes
