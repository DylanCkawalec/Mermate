## Mission
Use the provided problem statement and materials to build a serious architecture through iterative Mermate reasoning.

The goal is not to produce a quick diagram.
The goal is to produce a correct, refined, useful architecture that reflects the actual problem and is strong enough for engineering discussion.

## Required operating behavior
1. First understand the problem being solved.
2. Classify the user input:
   - vague idea
   - developing architecture
   - strong architecture prose
   - markdown notes
   - mixed supporting material
3. Extract:
   - actors
   - services
   - stores
   - constraints
   - flows
   - failure paths
   - policy gates
   - state or lifecycle logic
   - missing but implied architecture
4. Create an initial architecture draft in Mermate language.
5. Iterate through copilot suggestions and selective enhancement.
6. Wait for the 3+ second suggestion cycle where appropriate.
7. Accept only useful suggestions.
8. Run repeated cheap/default renders to evaluate the emerging design.
9. Critique each diagram using the architecture rubric.
10. Split the architecture into multiple views if one view becomes too dense.
11. Strengthen the final text before Max mode.
12. Use Max mode only for final architect-grade output.

## System capabilities (active in pipeline)
The system has the following capabilities that operate automatically during inference:
- **Multi-role planning:** Up to 3 domain-expert agents reason concurrently on the initial plan. The best plan wins by combined quality + completeness score.
- **Graduated refinement:** Refinement pressure is gated by quantitative scoring — heavy rewrite below 0.5, targeted gap-fill between 0.5-0.7, skip above 0.7.
- **2-branch composition:** The render pipeline generates two structurally competing Mermaid compositions in parallel (TB/boundary vs LR/data-flow), scores both, and keeps the stronger candidate.
- **GoT-style merge:** If both composition branches survive pruning, compatible entities from the runner-up are merged into the winner when doing so does not decrease the score.
- **Per-stage model routing:** Structured extraction tasks use fast models (gpt-4.1-mini), creative composition uses worker models (gpt-5.2), and orchestration uses the orchestrator tier (gpt-5.4).
- **Per-stage token capping:** Each pipeline stage has an explicit max_completion_tokens budget to reduce waste.
- **Render audit events:** The full render pipeline emits granular audit events (fact extraction, plan, composition, validation, repair) for downstream narration and debugging.
- **Parallel sub-view decompose:** When a diagram is split into multiple views, all sub-views are rendered concurrently.
- **State budget enforcement:** The GoT controller tracks state count against a budget (default 40) and halts branching/repair if the budget is reached.

Leverage these capabilities: trust the multi-role planning for initial breadth, focus your refinement on the gaps the scorer identifies, and use the 2-branch composition to explore genuinely different layout strategies.
