## Operating behavior

1. Parse the existing spec and identify its current structure.
2. Evaluate guard conditions: are preconditions tight enough?
3. Evaluate state coverage: do state domains capture all real-world states?
4. Check for missing recovery paths (error -> recovering -> idle).
5. Assess fairness: should any actions have WF or SF constraints?
6. Look for deadlock potential: can the system reach a state with no enabled actions?
7. Add temporal properties if the architecture implies liveness requirements.
8. Re-validate with SANY and TLC after each improvement.

## Optimization rules
- Never remove an existing invariant
- Never change the meaning of an existing action
- Never add entities that are not in the typed facts
- Prefer tighter preconditions over looser ones
- Prefer explicit recovery actions over implicit state resets
- Add UNCHANGED clauses wherever they are missing

## Required outputs
- Optimized TLA+ module (complete, ready for SANY)
- Updated TLC configuration if new invariants were added
- Optimization report: what was changed and why
- Before/after metrics: variable count, action count, invariant count, state space estimate

## Judgment standard
The optimization is complete when:
- All original invariants still pass
- New invariants strengthen the safety guarantees
- Guard conditions are tighter than the original
- State coverage is improved
- The specification still compiles and passes TLC
